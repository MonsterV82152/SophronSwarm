## 0. Locked decisions (operator-approved 2026-07-12)

| # | Decision | Rationale |
|---|---|---|
| **D1** | **G_O is a frontier agent** with an inline `propose_roster` skill. The separate `architect` agent is **removed entirely**. | The G_O has full conversation context (goal, requirements, vision, scale) — it should design the architecture itself. A second model = configuration mess + agent-to-agent information loss. |
| **D2** | **Every agent requires BOTH `provider:` AND `model:`** in frontmatter. No built-in defaults, no `defaultModel` on providers, no `inherit` tier, **no tier system at all** (tiers removed entirely). `sophron init` forces provider setup. | "No 1 model fits all scenarios" + frontier models constantly change, so a fixed "frontier" mapping is stale on arrival. Concrete model ids only. |
| **D3** | **Layered chrome**: chat views render bare (no ASCII banner, no bounding box — terminal expands naturally); dashboard views keep the boxed M3 chrome. | Feasible as a per-view `chrome` mode; preserves the 33 nav tests and the tested dashboard. |
| **D4** | **Every agent gets a "channel"** — a live view of what it's doing (stream). You can enter any running agent's channel, observe its activity in real time, **stop** it if something goes wrong, and `/model` it. Orchestrators (G_O + per-project) are also **interactive** (you can chat with them + `@file` mentions). Worker agents are **observation-only** (stream + stop + `/model`, no chat). The tabbed dashboard is retained for cross-agent monitoring. | Observability + control of the swarm is the core need. Not every agent needs two-way chat, but every running agent needs to be watchable and stoppable. |

## 2. What's removed in V3.1.0

Removing things first (explicitly) prevents dead code from accumulating:

| Removed | Why | Replacement |
|---|---|---|
| `builtinDefaults()` + `mergeWithDefaults()` in providers.ts | D2: no silent defaults | Operators must configure providers explicitly |
| `defaultModel` field on `ProviderConfig` + `RawProviderEntry` | D2: no default model | Agents declare `model:` explicitly |
| The ENTIRE tier system: `ModelTier` type (`inherit`/`frontier`/`mid`/`cheap`), `tiers` config in `OperatorConfig`, tier-resolution logic in `resolveModel` | D2: tiers removed entirely — frontier models change constantly, a fixed mapping is stale on arrival | `model:` is always a concrete model id string |
| Env-var model fallbacks (`OLLAMA_DEFAULT_MODEL`, `OPENROUTER_DEFAULT_MODEL`, `ZAI_DEFAULT_MODEL`) | D2: no silent defaults | Operators configure providers explicitly via `sophron providers add` |
| `GLOBAL_ARCHITECT` template + `installGlobalArchitect()` | D1: architect removed | G_O does architecture inline |
| `architect` from G_O `delegateAllowlist` | D1: no architect | G_O calls `propose_roster` directly |
| `--install-architect` CLI flag | D1: architect removed | `--install-orchestrator` covers the G_O |
| Top-level `add-provider` / `remove-provider` / `edit-provider` commands | Consolidation (prompt §Providers) | `sophron providers add` / `remove` / `edit` subcommands |

## 3. Milestone V3.1.0-M1 — Provider + Model Contract Refactor

**Goal:** Make `provider:` + `model:` mandatory; remove all silent defaults; add
provider `description` as a first-class field. This is the foundation — every
other milestone reads the new contract.

**`resolveModel`** — simplified, both args required, no tier indirection:
```typescript

  // 1. Validate the provider exists (throws clear error if not).
  const inst = getProvider(provider);
  // 2. Return the model as-is (always a concrete id — tiers are removed).
  return { provider: inst.name, model };
```

becomes a plain `string` (concrete model id). The `modelTier` field on


## 6. Milestone V3.1.0-M4 — Agent Channels: Live Stream + Interrupt + Interactive Chat

**Goal:** Every running agent has a **channel** — a live, real-time view of what
it's doing. The operator can enter any agent's channel to observe activity,
**stop** it mid-run, and `/model` it. Orchestrators (G_O + per-project) are
also **interactive** (chat with `@file` mentions + autocomplete). Worker agents
are **observation-only** (stream + stop + `/model`, no two-way chat).

**Depends on:** V3.1.0-M3 (chrome layering provides the bare rendering surface).

### 6.1 The two interaction modes

| Surface | Who | Input | Capabilities |
|---|---|---|---|
| **Interactive channel** | Orchestrators (G_O, per-project orchestrator) | `ChatInput` (enabled) | Send messages, `@file` attachments, `/` + `@` autocomplete menus, `/model`, observe live stream, stop |
| **Observation channel** | Worker agents (builder, tester, any delegated sub-agent) | `ChatInput` (disabled / hidden) | Observe live stream, `/model`, stop. No sending messages. |

Both surfaces share the **same rendering pipeline** (a `ChannelView` component)
— the only difference is whether the input bar is interactive.

### 6.2 Architecture

```
ChannelView (per-agent, bare chrome)
  ├── Status line: agent name · model · status (running/idle/stopped) · turn N
  ├── Activity thread (unified: chat messages + tool-call summaries + LLM turns)
  │     ├── user: "implement the auth module"          (interactive only)
  │     ├── 🧭 › Delegating to builder...               (agent response)
  │     ├── 🔧 builder › turn 3 · run_command: npm test  (live sub-agent activity)
  │     ├── 🔧 builder › ✓ exit 0 · 15 passed            (purified tool result)
  │     ├── 🔧 builder › turn 4 · write_file: src/auth.ts
  │     └── 🧭 › Builder reports auth module complete.
  ├── Controls: [Ctrl+C / /stop] stop  [/model] change model  [Esc] back
  └── ChatInput (interactive channels only; observation channels show "(read-only)")
```

### 6.3 Contract changes

#### `src/agent/events.ts` (NEW — the real-time event bus)

The piece deferred from M3. A lightweight in-process `EventEmitter` the loop
emits to alongside the JSONL recorder. This is what enables **real-time**
streaming (the M3 AgentDetail polled JSONL every 500ms — too slow for live
observation).

```typescript
import { EventEmitter } from "node:events";

export interface AgentEvent {
  runId: string;
  agentName: string;
  type: "run_start" | "turn_start" | "llm_response" | "tool_call_start" | "tool_call_result" | "run_end" | "run_error";
  turn?: number;
  // type-specific fields (tool name, result preview, usage, status, etc.)
  [key: string]: unknown;
}

/** Global event bus. The loop emits; the TUI subscribes. */
class AgentEventBus extends EventEmitter {
  emit(event: AgentEvent): void { super.emit(event.runId, event); super.emit("*", event); }
  onRun(runId: string, handler: (e: AgentEvent) => void): () => void {
    super.on(runId, handler);
    return () => super.off(runId, handler);
  }
}
export const agentEvents = new AgentEventBus();
```

**Key design:** events are keyed by `runId` so a channel subscribes to exactly

#### `src/agent/loop.ts` — emit events + abort support

1. **Emit alongside every `recorder.record(...)` call.** Every existing
   recorder call gets a parallel `agentEvents.emit(...)` with the same data.
   This is mechanical — there are ~8 emit points (run_start, turn_start,
   llm_response, tool_call_start, tool_call_result, turn_end, run_end,
   run_error).

2. **Add `AbortSignal` support:**
```typescript
export interface RunOptions {
  // ... existing fields ...
  /** Optional abort signal. The loop checks this between turns and aborts
   *  long-running tool commands. When aborted, the run ends with status "stopped". */
  abortSignal?: AbortSignal;
}
```
   - **Between turns** (top of the `for` loop, line ~124): check
     `abortSignal?.aborted` → if true, set `state.status = "stopped"`, break.
   - **During tool dispatch:** thread the signal into `dispatcher.dispatch` →
     `run_command` → `spawnWithTimeout` (which already has AbortController
     support from Phase 1). When aborted mid-tool, the tool returns an error
     result ("aborted by operator"), the loop sees the signal on the next
     check and stops.

3. **Add `state.status = "stopped"`** to the `AgentRunState.status` union
   (currently `"running" | "complete" | "halted" | "error"`). This is distinct
   from "halted" (maxTurns) and "error" (exception) — it's an intentional
   operator stop.

### 6.6 Risks / gotchas

- **EventEmitter in tests:** `agentEvents` is a module-level singleton. Tests must clean up listeners (`onRun` returns an unsubscribe function — use it in `afterEach`). Or create a fresh bus per test (inject rather than import the singleton). Recommendation: the singleton is fine for production; tests use the unsubscribe return.
- **Abort + delegation:** when a parent agent is stopped, its delegated sub-agents should also stop. The parent's abort signal must propagate to child `runAgent` calls. The `delegate` tool creates a child run — it should forward the parent's abort signal (or the `RunManager` tracks the parent-child relationship and stops children when the parent is stopped). **Implement this in Phase A** — add `parentId` to `ActiveRun` and cascade stop.
- **Observation of delegated sub-agents:** when the orchestrator delegates to the builder, the builder's run has its own `runId`. The orchestrator's channel shows a summary ("delegating to builder…"); to see the builder's live activity, the operator navigates to the builder's channel. The dashboard's agent list shows all active runs so the operator can find the builder's channel. **This requires the `RunManager.listActive()` to be surfaced in the TUI** — the Agents tab (boxed dashboard) shows running agents with an indicator; Enter opens the channel.
- **Menu vs nav reducer conflict:** same as original M4 — `ChatInput` swallows all keys when focused. The nav reducer's input-focus guard (`if (state.focus === "input") return`) already handles this.
- **`interactive` determination:** how does the TUI know if a channel is interactive? Rule: an agent is interactive if it has `delegate` in its tools (it's a coordinator) OR if it's the per-project orchestrator / G_O. Worker agents (no `delegate`) are observation-only. This can be derived from the `AgentDefinition.tools` list. **Alternative:** add an `interactive: true` frontmatter flag for explicitness. Recommendation: derive from `tools.includes("delegate")` for now; add the flag if operators want to override.

## 8. Cross-feature invariants (prevent AI-slop)

These invariants must hold across ALL milestones. Any feature that violates
them is wrong:

| # | Invariant | Enforced by |
|---|---|---|
| **I1** | `resolveModel(model, provider)` is the SINGLE model-resolution function. Both args required. No fallbacks. | `providers.ts` — no other function resolves models. |
| **I2** | Every `AgentDefinition` has a non-empty `model` AND `provider` after loading. | `FrontmatterSchema` (zod required fields) + loader. |
| **I3** | All agent-channel rendering (`ChannelView` + `OrchestratorChat`) shares ONE `MessageThread` component. No rendering divergence. | Shared `MessageThread` component (M4 step 7). |
| **I4** | Dashboard navigation is owned by `nav.ts` (pure reducer). Chat input micro-interactions are owned by `ChatInput`. They never share `useInput` handlers. | Separation of concerns; `app.tsx` routes keys to one or the other based on focus. |
| **I5** | Tool output entering a channel thread is ALWAYS purified (M1 output purifier). Raw output is available via `read_raw_output`, never dumped into context. | The channel's event handler shows purified summaries; raw goes to `.sophron/raw/`. |
| **I6** | The G_O has NO memory, NO workspace, NO `delegate`, NO `run_command`. It designs rosters inline via `propose_roster`. | `GLOBAL_ORCHESTRATOR` template + `noMemory: true` + tool allowlist. |
| **I7** | Provider config changes (add/edit/remove) ALWAYS reset the lazy cache so the running process reflects them. | `_resetProviderCacheForTests()` called after every `writeRawConfig`. |
| **I8** | Stopping a parent agent cascades to all its delegated children. No orphaned sub-agent runs. | `RunManager` tracks `parentId`; `stop(runId)` aborts the run + all children recursively. |
| **I9** | `model:` is ALWAYS a concrete model id string. No tiers, no `inherit`, no indirection. | `FrontmatterSchema` (`z.string().min(1)`) + `resolveModel` returns the model as-is. |
| **I10** | An agent channel is `interactive` iff the agent has `delegate` in its tools (it's a coordinator). Worker agents are observation-only. | `ChannelView` derives `interactive` from `agent.tools?.includes("delegate")`. |
