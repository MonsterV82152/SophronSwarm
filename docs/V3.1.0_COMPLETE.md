## 1. Executive Summary

V3.1.0 refactored SophronSwarm around a strict provider+model contract, consolidated the CLI surface, and upgraded the operator experience with real-time agent channels and a first-run configuration wizard.

The four headline changes:

1. **Provider + model contract (M1):** Removed the entire tier system, removed built-in model defaults, and made `model:` (concrete id) and `provider:` (configured instance name) mandatory on every agent. `resolveModel(model, provider)` is the single resolution chokepoint.
2. **Global Orchestrator consolidation (M2):** Removed the separate `architect` agent; the Global Orchestrator now designs rosters inline using `propose_roster`, `propose_agent`, and `list_providers`. It remains scope-limited (`noMemory: true`, no `delegate`, no workspace mutation tools).
3. **TUI chrome layering + agent channels (M3/M4):** Chat surfaces render in bare chrome; dashboards stay boxed. Every running agent has a live `ChannelView` backed by `AgentEventBus` and `RunManager` with cascading abort. Orchestrator channels are interactive (`ChatInput` with `@file` attachments); worker channels are observation-only.
4. **CLI consolidation + init wizard (M5):** Provider management moved under `sophron providers <subcommand>`. `sophron init` runs an interactive provider wizard when no providers exist. `sophron agents edit <name> --model/--provider` mutates frontmatter through `resolveModel()`.

## 3. Milestone-by-Milestone Change Log

### 3.1 V3.1.0-M1 — Provider + Model Contract Refactor

**Goal:** Make `model:` and `provider:` mandatory, remove all silent defaults, and delete the tier system.

**Key changes:**

- `src/llm/providers.ts`
  - Removed `ModelTier`, `modelTier`, `defaultModel`, `tiers{}`, `builtinDefaults()`, `mergeWithDefaults()`, `defaultForKind()`, `resolveModelWithProvider()`.
  - `ProviderConfig` gained `description` and lost `defaultModel`.
  - `OperatorConfig` lost `tiers`.
  - `resolveModel(model, provider)` now requires both arguments and returns the model as-is after validating the provider instance.
  - Added `reresolveModel(agent, newModel?, newProvider?)` for runtime `/model` overrides.
  - `loadInstances()` returns `[]` when no providers are configured (no built-in singletons).

- `src/types.ts`
  - Removed `ModelTier` type and `modelTier` field from `AgentDefinition`.
  - Added `noMemory?: boolean` (used by M2 Global Orchestrator).

- `src/agent/loader.ts`
  - `FrontmatterSchema.model` changed from tier enum/`inherit` to `z.string().min(1)` (required concrete id).
  - `FrontmatterSchema.provider` changed from optional to required.
  - Replaced `resolveModelWithProvider()` with `resolveModel()`.
  - Added `updateAgentFrontmatter(filePath, { model?, provider? })` for targeted disk updates.
  - Later (M5 bugfix) changed `matter(raw)` → `matter(raw, {})` to disable gray-matter's body-content cache when callers mutate `parsed.data`.

- `src/llm/client.ts`
  - `complete()` now requires `req.provider` and throws a clear error if it is missing.

- `src/tui/app.tsx` (M3/M4 additions built on this contract)
  - `/model` command calls `reresolveModel()` and persists the result via `updateAgentFrontmatter()`.

**Tests:** `tests/llm/providers.test.ts`, `tests/agent/loader.test.ts`, `tests/init/templates.test.ts`.

### 5.2 Issues Found and Fixed in This Review

#### Fix 1: `serializeDraft` could emit `model: inherit` and omitted provider validation

**Location:** `src/agent/serialize.ts:34-41`

**Problem:** `serializeDraft()` fell back to `model: inherit` when the LLM omitted a concrete model. This violated the V3.1.0 invariant that `model:` must always be a concrete id string. A draft with `model: inherit` would fail to load because `FrontmatterSchema.model` requires `z.string().min(1)`. The function also silently omitted `provider:` when it was missing, producing equally unloadable drafts.

**Fix:** Replaced the fallback with explicit validation that rejects missing or empty `model` and `provider`.

```typescript
// Before
if (typeof f.model === "string") fm.push(`model: ${yamlString(f.model)}`);
else fm.push("model: inherit");
if (typeof f.provider === "string") fm.push(`provider: ${yamlString(f.provider)}`);

// After
if (typeof f.model !== "string" || !f.model.trim()) {
  throw new Error("Draft agent must have a concrete model id (V3.1.0 no longer supports 'inherit').");
}
fm.push(`model: ${yamlString(f.model)}`);
if (typeof f.provider !== "string" || !f.provider.trim()) {
  throw new Error("Draft agent must have a configured provider name.");
}
fm.push(`provider: ${yamlString(f.provider)}`);
```

**Test updates:** Existing tests for `serializeDraft`, `propose_agent`, and `propose_roster` were still using the pre-V3.1.0 tier values (`model: "inherit"`, `model: "mid"`) or omitting model/provider entirely. All were updated to supply concrete `model` + `provider` pairs so the suite remains green.

**Impact:** `propose_agent` and `propose_roster` now fail fast if the LLM forgets to specify a concrete model and provider, rather than producing unloadable drafts.

#### Fix 2: Non-interactive `providers add --kind openai-compat` allowed missing `--base-url`

**Location:** `src/cli/providers.ts:111-115`

**Problem:** The interactive path required a base URL for `openai-compat`, but the non-interactive path did not. This allowed creation of an invalid provider with `baseURL: ""`, which would fail at request time.

**Fix:** Added validation so `openai-compat` requires `--base-url` in non-interactive mode.

```typescript
if (nonInteractive && kind === "openai-compat" && !opts.baseUrl) {
  console.error(chalk.red("--base-url is required for kind 'openai-compat'."));
  process.exitCode = 1;
  return;
}
```

**Impact:** `sophron providers add --kind openai-compat` now fails immediately with a clear message instead of creating a broken config.

### 5.3 Issues Found and Fixed in the Quality Pass (2026-07-14)

The following issues were identified during the post-M5 review and fixed:

#### Q1: `listFiles` walked the entire workspace on every `@` keystroke (FIXED)

**Location:** `src/tui/components/ChatInput.tsx`

**Problem:** `listFiles()` called `readdirSync` recursively across the entire workspace every time the user typed a character after `@`. No caching, no directory filtering (only skipped `.`-prefixed entries), no depth limit. This caused noticeable lag in large workspaces.

**Fix:** Added a module-level cache with a 5-second TTL per workspace root, a `SKIP_DIRS` set (`node_modules`, `.git`, `dist`, `build`, `.next`, `coverage`, etc.), a max walk depth of 15, and a max file count of 5000.

#### Q2: Stale JSDoc in `providers.ts` referenced removed `defaultModel` (FIXED)

**Location:** `src/llm/providers.ts` (lines 22-24, 30-39)

**Problem:** The header docstring example config still showed `"defaultModel": "qwen3.5:9b"` on each provider entry, and the "Backward compatibility" section described zero-config built-in singletons — both removed in M1.

**Fix:** Rewrote the example config to omit `defaultModel`, replaced the backward-compat section with a "V3.1.0 changes" section, and removed the stale "built-in singletons" reference from the `ProviderName` type comment.

#### Q3: `ChannelView` lost events published before subscription (FIXED)

**Location:** `src/agent/events.ts`, `src/tui/components/ChannelView.tsx`

**Problem:** `AgentEventBus` was a pure `EventEmitter` with no replay buffer. When the operator entered a channel after a run started, earlier events were lost.

**Fix:** Added a 500-event ring buffer to `AgentEventBus`. New `recentForAgent(agentName, limit?)` returns recent events for catch-up. `ChannelView` seeds its `items` state from the buffer on mount, then subscribes to live events.

#### Q4: Delegated children not registered in `RunManager` (FIXED)

**Location:** `src/tools/builtin/delegate.ts`, `src/agent/runManager.ts`

**Problem:** `delegate.ts` called `runAgent()` directly, bypassing `RunManager.start()`. This meant delegated children weren't visible in `listActive()` and the `parentId` cascade in `stop()` was effectively dead code for real delegations.

**Fix:** `delegate.ts` now calls `runManager.start()` with `parentId: state.runId` and `parentSignal: signal`. `RunManager.start()` gained a `parentSignal` option that wires the parent's abort signal to the child's `AbortController` (in addition to the `parentId` cascade). Both registration paths ensure delegated children are tracked and cascade-stopped.

#### Q5: Dead `default` provider flag (FIXED)

**Location:** `src/llm/providers.ts`, `src/cli/providers.ts`, `src/init/wizard.ts`

**Problem:** The `default` boolean flag on provider entries was stored, displayed in `providers view`, editable via `--default`/`--no-default`, and prompted for interactively — but M1 removed `defaultForKind()` and prefix-shortcut resolution, so the flag was never read.

**Fix:** Removed `default` from `AddProviderInput`, `ProviderPatch`, CLI options (`--default`/`--no-default`), interactive prompts, and `providers view` output. Kept it in `RawProviderEntry` (marked vestigial) so existing configs with `default: true` don't cause parse errors — it's silently ignored on read.

#### Q6: `@(\S+)` regex couldn't match file paths with spaces (FIXED)

**Location:** `src/tui/components/ChatInput.tsx`

**Problem:** The `@(\S+)` regex in `resolveAttachments()` stopped at the first whitespace, so `@file with spaces.ts` was parsed as `@file`.

**Fix:** Extended the regex to `@"([^"]+)"|@(\S+)` — supports quoted syntax `@"path with spaces.ts"` in addition to the existing bare-path syntax.
